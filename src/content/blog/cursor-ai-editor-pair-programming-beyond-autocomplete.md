你有沒有遇過這種狀況：在 IDE 裡卡了一個 bug 半小時，最後還是去 Stack Overflow 找答案，然後又花時間把解法搬回編輯器？

Cursor 想改變這件事。

## Cursor 是什麼？

Cursor 是一款以 AI 協作為核心設計的程式碼編輯器。它的底層基於 VS Code，所以對已經習慣 VS Code 的開發者來說，切換成本極低——外掛、快捷鍵、介面幾乎一模一樣。

但最大的不同在於：Cursor 把 AI 直接整合進編輯流程，而不是以附加外掛的方式加上去。

---

## 三個讓 Cursor 和一般補全工具不同的功能

### 1. Cmd+K：在程式碼裡直接對話

你可以選取一段程式碼，按下 Cmd+K，然後直接用自然語言告訴 AI 你想做什麼：「把這個函式改成非同步的」、「加上錯誤處理」、「用更簡潔的方式重寫」。AI 會在原本的位置直接修改，你接受或拒絕就好。

這和把程式碼貼到 ChatGPT 再複製回來的流程是完全不同的體驗。

### 2. Chat 視窗：帶著整個 codebase 問問題

Cursor 的 AI 對話不只看你目前開啟的檔案，它可以索引整個專案，讓你問出像「這個 API 是在哪裡被呼叫的？」或「我想新增一個使用者登入功能，從哪裡開始？」這樣的問題，並得到有根據的回答。

對於剛接手舊專案或大型 codebase 的開發者，這個功能特別有價值。

### 3. Composer：從描述到完整程式碼

Composer 是 Cursor 更進一步的功能——你描述你想建立的東西，它會跨越多個檔案生成對應的程式碼結構。不是單一函式的補全，而是更接近「從需求到實作」的過程。

## Cursor 適合誰？

**最適合的使用情境：**
- 獨立開發者或小團隊，沒有大量 code review 資源
- 需要快速迭代、頻繁修改邏輯的專案
- 剛接手不熟悉的 codebase
- 想減少在文件和 Stack Overflow 之間來回切換的時間

**可能不是最佳選擇的情況：**
- 團隊有嚴格的工具統一要求
- 對程式碼由 AI 修改有安全疑慮的企業環境（需要評估隱私政策）

## 和 GitHub Copilot 的差異在哪？

GitHub Copilot 主要強項在於行內補全，它非常擅長預測你接下來要寫什麼。Cursor 的定位則更偏向「對話式開發夥伴」——除了補全，還提供跨檔案的對話、重構協助、以及更接近自然語言需求描述的開發流程。

兩者並非完全互斥，但如果你在找的是能真正「討論程式碼」而不只是「預測下一行」的工具，Cursor 的設計哲學更接近這個方向。
